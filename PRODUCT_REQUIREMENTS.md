# PT Manager 2.0 产品需求文档 (PRD)

> **文档状态**：草案
> **产品名称**：PT Manager 2.0 (重构版)
> **技术栈**：Tauri v2 + React (shadcn/ui) + Rust

## 1. 产品概述

### 1.1 产品定位
PT Manager 是一款专为 PT (Private Tracker) 玩家设计的轻量级、自动化站点保活客户端。它能够在后台静默运行，定期自动访问已配置的 PT 站点，维持登录活跃状态，避免账号因长期未登录而被系统封禁。

### 1.2 重构背景
原产品基于 **Vue3 + Electron** 开发，但在实际使用中暴露出了明显的痛点：
1. **体积与性能臃肿**：Electron 强行捆绑了庞大的 Chromium 引擎，导致安装包高达 150MB，且常驻后台内存消耗过大（~200MB），不符合“轻量级挂机工具”的定位。
2. **对抗 Cloudflare 盾（Turnstile）乏力**：Electron 的内置 BrowserWindow 缺乏用户日常主浏览器的真实使用轨迹和 Cookie，极易被 CF 判定为机器行为并拦截，导致保活失败。

本次重构全面拥抱 **React + Tauri v2** 技术栈，从架构底层彻底解决上述痛点，实现极致轻量化与真正无感的静默保活。

---

## 2. 目标用户与痛点分析

### 2.1 目标用户
- 持有多个 PT 站账号的重度/中度玩家。
- 无法保证每天手动逐个登录所有站点，需要自动化工具代劳的用户。

### 2.2 核心痛点
- **频繁验证掉线**：各大 PT 站广泛部署 Cloudflare，常规的无头浏览器或脚本工具被拦截率极高。
- **后台资源占用**：作为 24 小时运行的工具，若大量占用 CPU 和内存，会影响用户电脑的日常工作和游戏体验。
- **配置门槛**：过于硬核的脚本工具对小白不友好，需要图形化界面的傻瓜式操作。

---

## 3. 核心功能需求

### 3.1 CDP 静默接管引擎（核心破盾特性）
产品不再内置浏览器环境，而是通过 **CDP (Chrome DevTools Protocol) 的 REST API** 机制，直接“借用”用户本机日常使用的 Chrome 浏览器。
- **工作机制**：在设定的保活时间，后台静默向本地 Chrome 发送指令（`PUT /json/new`）打开新标签页。由于直接复用了用户的真实浏览器环境、指纹以及已通过验证的有效 Cookie（含 `cf_clearance`），Cloudflare 会完全放行，无需任何人工介入。
- **无感体验**：页面加载停留数十秒（可配置）后，再次通过 CDP 指令（`DELETE /json/close`）关闭该标签页。

### 3.2 站点与配置管理
- **站点列表**：支持新增、编辑、删除需要保活的站点名称及 URL。
- **参数设置**：
  - 支持自定义 CDP 端口（默认 9222）。
  - 支持设置页面访问的停留时长（以确保 Cookie 完全刷新）。

### 3.3 自动化与防风控调度
- **Cron 表达式调度**：内置基于 Rust 的 `tokio-cron-scheduler`，支持灵活的定时任务配置（例如每天上午 9 点执行）。
- **拟人化随机抖动**：支持开启“随机延迟”，在设定的执行时间点前后引入一定的随机延迟（如 ±30 分钟），且不同站点间的访问也加入随机间隔（5~15秒），避免死板的定时请求引起服务器风控。
- **开机自启**：支持随操作系统开机自动启动，常驻后台。

### 3.4 状态监控 Dashboard (新特性)
- 实时可视化展示 **Chrome CDP 连接状态**（绿灯正常 / 红灯断开）。
- 显示下一轮保活任务的预计执行时间。
- 展示最近一次的任务执行结果状态。
- 一键手动触发“立即执行”功能。

### 3.5 运行日志
- 实时滚动展示保活任务日志（成功、失败、错误原因），方便用户排查站点失效情况。

## 4. 界面与交互体验设计

### 4.1 UI 设计语言
- 采用 **shadcn/ui** 组件库，搭配 Tailwind CSS，打造现代、极简的桌面端应用体验。
- 默认支持并推荐 **深色模式 (Dark Mode)**，符合极客与技术用户的审美。

### 4.2 核心交互流程
1. **初次启动引导**：提示用户修改本机的 Chrome 快捷方式（追加 `--remote-debugging-port=9222`）并重启浏览器。系统检测到 CDP 连通后，给出连接成功的视觉反馈。
2. **日常待机**：主界面可通过右上角或菜单关闭，程序不退出，而是**最小化至系统托盘 (System Tray)**。
3. **托盘交互**：托盘提供鼠标悬停状态提示（Tooltip），右键菜单提供：“打开主界面”、“立即执行保活”、“退出”选项。双击托盘图标可快速唤醒主界面。

## 5. 非功能性需求 (非功能指标)

| 指标维度 | 要求 / 目标 | 说明 |
| --- | --- | --- |
| **包体积** | **< 10 MB** | 摒弃 V8 引擎，完全依靠系统原生 WebView 构建 UI。 |
| **内存占用** | **< 50 MB** | Rust 后端的高效内存管理，保证后台常驻极低损耗。 |
| **安全性** | 高 | CDP 接口调用仅限本地 `localhost`，不向公网暴露任何接口；前端与系统底层隔离，采用安全的 Tauri IPC invoke 通信。 |
| **平台兼容** | 初期主打 Windows | 针对 Windows 用户的 Chrome 快捷方式使用习惯优化。未来可扩展 macOS。 |

---

## 6. 产品演进 Roadmap

- **v2.0 (MVP 阶段)**：完成 Tauri + React 重构，跑通 CDP 静默打开关闭标签页的核心链路，实现基础定时任务、托盘和日志功能。
- **v2.1 (数据增强)**：支持从保活页面中抓取站点的基础信息（如魔力值、做种数、上传下载量），并在 Dashboard 中可视化汇总展示。
- **v2.2 (多浏览器支持)**：拓展 CDP 接管能力，支持 Edge 浏览器或其他基于 Chromium 的浏览器。
- **v2.3 (消息推送)**：集成 Server酱、Telegram Bot 等渠道，在 Cookie 失效需要人工介入时，发送推送通知。
